# Investigation Report: Google Photos Backup Failure for iPhone HEIC Images

This report investigates why Google Photos successfully backed up `IMG_6008.HEIC` but failed to back up `IMG_6009.HEIC` and `IMG_6010.HEIC`.

## Summary of Findings

The three images are structurally very similar HEIC files taken with an iPhone 17. While they appear almost identical in terms of resolution and container structure, a key difference was found in the camera hardware used and the resulting exposure metadata.

| Feature | IMG_6008 (Success) | IMG_6009 (Fail) | IMG_6010 (Fail) |
| :--- | :--- | :--- | :--- |
| **Lens/Camera** | Dual Ultra Wide | Dual Wide | Dual Wide |
| **Focal Length** | 14mm (equiv) | 26mm (equiv) | 26mm (equiv) |
| **Aperture** | f/2.2 | f/1.6 | f/1.6 |
| **Resolution** | 5712x4284 (24MP) | 5712x4284 (24MP) | 5712x4284 (24MP) |
| **File Size** | 4.39 MB | 4.23 MB | 4.22 MB |

## Detailed Analysis

### 1. Metadata (EXIF) Comparison
Using a custom EXIF analysis script, I compared the metadata of the three files.
- **Hardware Difference:** `IMG_6008` was taken with the "Ultra Wide" camera (14mm equivalent), while `IMG_6009` and `IMG_6010` were taken with the "Wide" camera (26mm equivalent).
- **MakerNotes:** The images contain Apple-specific MakerNotes. `IMG_6008` has some unique tags (e.g., `Tag 0x0031`, `Tag 0x0032`, `Tag 0x005C`) that are missing in the other two. Conversely, `IMG_6009/10` have tags like `Tag 0x000C`, `Tag 0x000D`, and `Tag 0x0010` which are missing in `6008`.
- **Timestamps:** The photos were taken only seconds apart (14:32:18, 14:32:20, 14:32:22).

### 2. HEIF Container Structure
I developed a script (`dump_heif.py`) to parse the internal "box" structure of the HEIC files.
- **Identical Structure:** All three files follow the standard HEIF structure: `ftyp` -> `meta` (containing `hdlr`, `pitm`, `iloc`, etc.) -> `mdat`.
- **Auxiliary Data:** All three files contain an `auxC` box, which typically points to auxiliary images like depth maps or alpha planes.
- **Color Profiles:** There is a slight size difference (12 bytes) in one of the `colr` boxes in `IMG_6008` compared to the others, suggesting a slightly different ICC profile or color metadata for the Ultra Wide lens.

### 3. File Integrity
- All files are complete and not truncated. The sum of the internal box sizes matches the total file size reported by the filesystem.

## Potential Reasons for Backup Failure

1.  **Server-Side Deduplication or Glitch:** Since the images are very similar and taken in rapid succession, Google Photos' deduplication logic might be incorrectly flagging `6009` and `6010` as duplicates or "near-matches" of `6008` (or each other) and failing to process them, though they should be treated as distinct.
2.  **Wide Lens Metadata Issue:** The "Wide" camera on this device (iPhone 17) might be producing a specific MakerNote or metadata combination that Google Photos' HEIC parser currently chokes on, whereas the "Ultra Wide" metadata is handled correctly.
3.  **ProRAW/High Resolution Processing:** Even though the reported resolution is 24MP for all, the "Wide" lens often uses different sensor modes (like 48MP binned to 24MP). If there's an internal "Gain Map" or HDR metadata specific to the Wide sensor that is slightly malformed or unsupported, it could cause the upload to hang or be rejected.

## Conclusion
The issue is likely not with the files themselves (as they are structurally sound) but with how Google Photos' backend processes the specific metadata generated by the "Wide" lens of this iPhone model compared to the "Ultra Wide" lens.
